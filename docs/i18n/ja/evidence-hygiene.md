# Evidence Hygiene（証拠衛生）

Agentic Patterns は独自ソリューションの詳細を公開せずに有用であるべきです。

カタログは実際の経験に基づくことができますが、非公開のプロンプト、ハーネス、
リポジトリ、顧客、認証情報、トランスクリプト、内部ワークフロー機構は保護する必要があります。

## 原則

パターンを公開し、非公開インシデントは公開しません。

例は以下の再利用可能な構造を示すべきです。

- 問題の形。
- Forces とトレードオフ。
- 手順。
- 期待される出力。
- 失敗モード。
- 検証スタイル。

例は非公開の実装詳細を明かすべきではありません。

## 安全な証拠の種類

- ワークフローの構造を保持した synthetic example。
- 非公開識別子を除去した sanitized フィールドノート。
- オープンソースリポジトリ、論文、ドキュメント、ブログ記事、issue スレッドからの公開例。
- 識別情報を含まない集約された観察。

## 公開してはいけないもの

- 顧客データ。
- 認証情報、トークン、または非公開 URL。
- プライベートリポジトリ名。
- 公開を意図していない内部プロンプトテキスト。
- 完全な独自ハーネスロジック。
- 非公開プロジェクトの完全なワークフロートランスクリプト。
- 製品方向性、ビジネス戦略、または未公開の実装詳細。

## フィールドノートのサニタイズ方法

1. プロジェクト名を `web app`、`API service`、`agent runner` などの一般的な名前に置き換えます。
2. ファイルパスを `src/billing/service.ts` などの代表的なパスに置き換えます。
3. 認証情報、ホスト名、非公開 issue 番号、非公開ブランチ名を除去します。
4. トランスクリプトを直接引用せずに要約します。
5. パターンに関連する成功・失敗条件を保持します。
6. 例が synthetic または sanitized であることを明記します。

## Synthetic Example ポリシー

Synthetic example は推奨されます。実際の非公開ワークフローに対応している必要はありませんが、
アクションを導くのに十分なほど現実的である必要があります。

良い synthetic example:

- 具体的なファイル、ロール、コマンドを含みます。
- パターンの forces を発揮します。
- エージェントが何を出力すべきかを示します。
- プロジェクト固有のシークレットを避けます。

悪い synthetic example は「コードをより良くする」のように曖昧すぎて実行できないものです。
