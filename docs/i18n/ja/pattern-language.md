# パターン言語

このドキュメントは Agentic Patterns エントリの構造と品質基準を定義します。

## 定義

Agentic pattern とは、human-AI 作業における反復する問題に対する、名前のついた再利用可能な解法です。

パターンは以下を説明できます。

- プロンプト構造。
- コンテキスト構造。
- 計画または実行のハーネス。
- レビューと検証のループ。
- エージェントのロール契約。
- MCP/ツールプロトコル。
- マルチエージェント調整の手法。
- メモリと状態管理の実践。

パターンはソフトウェアエンジニアリングに限定されません。Coding はテスト、リポジトリ、
ツール、レビューループが明確なため最初の強いドメインですが、同じ構造は画像生成、
執筆、デザイン、video/media production、リサーチ、分析、その他反復する
human-guided AI ワークフローにも適用されます。

## Pattern Metadata

各パターンは小さなメタデータブロックから始まります。

```yaml
---
id: context-packet
name: Context Packet
category: context
status: draft
applies_to:
  - coding agents
  - planning
  - implementation
domains:
  - coding
related:
  - durable-session-state
---
```

`id` は安定した lowercase のハイフン形式にする必要があります。`domains` は任意ですが
推奨されます。例: `coding`、`image-generation`、`writing`、`design`、`video-media`、
`research-analysis`、`general`。

## Pattern Sections

特別な理由がなければ以下のセクションを使用します。

### Intent

再利用可能な手法を 1〜2 文で説明します。

### Also Known As

別名があれば記載します。

### Problem

このパターンが解決する反復する問題です。

### Context

パターンが適用される状況。前提条件と環境の想定を含めます。

### Forces

問題を難しくする緊張とトレードオフです。例:

- コンテキストが多いほどエージェントを助けますが、多すぎると重要なシグナルが埋もれます。
- 自律性が高いほど実行が速くなりますが、無制限の自律性はリスクを隠す可能性があります。
- エージェントが多いほど作業を並列化できますが、調整のオーバーヘッドが利益を上回ることがあります。

### Solution

再利用可能な構造、プロトコル、プロンプト、ハーネス、またはツールの振る舞いです。

### Agent Recognition Signals

エージェントがこのパターンを検討すべき観察可能なシグナルです。

### Procedure

人間またはエージェントが従える具体的な手順です。

### Outputs

パターンが生成すべきアーティファクトです。

### Stop Conditions

いつパターンの適用を止め、エスカレートし、別のパターンに切り替えるかです。

### Example

コンパクトで現実的な例。実際のワークフローが非公開の実装、顧客、プロンプト、
オーケストレーションの詳細を明かす可能性がある場合は synthetic example を推奨します。

### Failure Modes

パターンが誤用される既知の方法です。

### Evidence

観察結果、フィールドノート、テスト、レビュー結果、その他の裏付けです。

証拠は公開に適している必要があります。sanitized な要約、synthetic な再現、または
公開ソースを使用し、非公開プロンプト、独自ハーネスロジック、顧客データ、
プライベートリポジトリ名、認証情報、完全な内部トランスクリプトを公開しないでください。

### Related Patterns

このパターンと組み合わせ、置換、または競合するパターンです。

## Pattern Status

- `draft`: 可能性はあるがまだ安定していない。
- `candidate`: 複数回成功裏に使用されレビュー準備ができている。
- `accepted`: レビューを経てカタログの一部として認定されている。
- `deprecated`: 歴史的参考のために残すが、もはや推奨しない。

## Categories

初期カテゴリ:

- `prompt`: 再利用可能な指示構造。
- `context`: 再利用可能なコンテキストパケット構造。
- `harness`: 計画、実行、レビュー、検証のための段階的ワークフロー。
- `orchestration`: マルチエージェントおよびマルチセッションの調整。
- `tool`: MCP、CLI、スクリプト、または環境インタラクションパターン。
- `memory`: 永続的な知識と状態パターン。
- `evaluation`: パターンが機能したかどうかを判断する方法。

カテゴリはカタログが成熟するにつれて変わる可能性があります。

## Evidence Hygiene（証拠衛生）

パターンの例は、パターンを生み出した非公開インシデントを開示するのではなく、
再利用可能な形を教えるものであるべきです。

許容される証拠:

- forces と procedure を保持した synthetic example。
- 非公開識別子を除去した sanitized フィールドノート。
- オープンソースリポジトリ、論文、ドキュメント、ブログ記事、issue スレッドなどの公開参照。
- 「複数のローカルコーディングセッションで観察された」などの集約された観察。

含めてはいけないもの:

- プライベートリポジトリ名、顧客名、非公開 URL、認証情報、ログ。
- 完全な独自プロンプト、ハーネス内部、または MCP/ツールプロトコル。
- 未公開の製品方向性や内部実装の詳細を明かすトランスクリプト。

疑わしい場合はシナリオを一般化し、パターンを operational（実用的）に保ちます。

## ネーミングルール

パターン名は以下であるべきです。

- 短い。
- 記憶しやすい。
- Operational（実用的）。
- 手法を区別するのに十分な具体性がある。

`Better Coding` や `Good Context` のように目標のみを表す名前は避けてください。

## 機械可読性

パターンファイルは通常の Markdown として有用であり続ける必要があります。
メタデータブロックは将来のメンテナーが以下を行えるように存在します。

- インデックスを構築する。
- 欠落フィールドを確認する。
- エージェントにパターンを推薦する。
